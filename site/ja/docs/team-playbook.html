チーム実践

Syfo をチームの日常業務に取り入れましょう

Channel、Agent、およびコンテキスト設計アプローチにより、個々のトライアルからチームのコラボレーションまで拡張できます。

チームが Syfo を正式に導入する準備をしているとき、最初のステップはAgent を構築することではなく、Channel を分割する方法、Agent の作業を分割する方法、およびコンテキストをどこに配置するかを理解することです。このガイドは、この計画を完了するのに役立ちます。

最初に本当に検討する必要があるのは、「いくつかのAgent を構築する」ことではなく、次のことです。

  1. どの事業分野がSyfoに参入すべきか。
  2. 各ビジネスラインをどのChannel に分割する必要があるか。
  3. 各 Channel には複数のAgent と、それらが果たす役割が必要です。
  4. Agent が実際に機能するためにはどのようなコンテキストを確認する必要がありますか?
  5. チームメンバーが遵守すべきコラボレーション規範。

まず事業ラインごとに Channel を分ける

Channelは Syfo 内の作業サイトです。Channelは、同じコンテキスト、同じ参加者のセット、同じTaskのセットの周りに存在する必要があります。

組織構造を最初からコピーしたり、すべてを詰め込んだりしないでください。 #general。まず、現在のチームが実行しているビジネスラインをリストアップし、次に独立したChannelが必要なビジネスラインを判断します。

シーン推奨Channelなぜこんなに解体されたのでしょうか？
お客様による実装#customer-a-implementation、#customer-a-delivery-execution顧客情報、配送の進捗状況、問題の追跡を一元化する必要があります。実装の詳細は、ディスカッション Channel から専用Channel に分割できます。
社内研究開発#product-planning、#dev-runtime、#release-review製品のディスカッション、コードの実装、およびオンライン検査には、さまざまな参加者とさまざまなAgent権限が必要です。
販売フォローアップ#sales-leads、#enterprise-demoセールスリード、顧客の背景、プレゼンテーション資料は、プライベートチャットに散在しないように継続的に収集する必要があります。
お客様のフィードバック#customer-feedback、#faq-improvements元のフィードバックとドキュメント/製品の改善は関連する可能性がありますが、同じスレッド内で混在させる必要はありません。

Channel を分割するときは、次の 4 つの次元を優先します。

  • プロジェクトまたはクライアント: 長期的なプロジェクト、顧客、配送ライン。独立したChannel に適しています。
  • 感度: 契約書、見積書、生産環境、顧客のプライバシー、社内担当者、その他の情報を公開Channel に公開しないでください。
  • 実行フェーズ: 要件の議論、計画のレビュー、実行の実施、およびオンライン レビューを分離して、1 つのChannelが長すぎたりノイズが多すぎたりするのを避けることができます。
  • 参加役割: 何かが長期間にわたって数人の人員と数人のAgentだけを必要とする場合、それを全員メンバーのChannel に入れないでください。

プロジェクトは 1 つのChannelまたは複数のChannel に対応できます。判断基準は単純です。同じグループの人々、同じAgent のグループ、および同じ権限セットが処理できる場合は、最初に 1 つのChannel に入れます。コンテキスト、権限、または実行リズムが明らかに異なる場合は、分割してください。

各 Channel に置く Agent 数を決める

Agent は多ければ多いほど良いです。Channel にAgent が追加されるたびに、同じメッセージがより多くのAgent によって読まれる可能性があり、ディスカッションが騒々しくなる可能性が高くなります。

まず次の判定方法を使用します。

Channel の種類推奨されるAgent の数役割に合う
デイリープロジェクトChannel常駐Agent 1 名コンテキストを整理し、Task を追跡し、責任者に思い出させ、明確な作業をTask に変換します。
特別な実行ChannelAgent1～2名1 人は実行の責任を負い、もう 1 人はレビューまたは承認の責任を負います。責任は違うはずだ。
攻撃と防御の計画/調査と議論短期Agent2～3名ユーザー、競合製品、エンジニアリング、データなどさまざまな視点から提案し、結論を出して結論を​​出します。
機密性の高いビジネスChannel少数の独立Agent権限に従って独立して構成し、他の顧客、部門、実稼働環境と混在させないでください。

Agent の役割を設計するときは、まず次の 3 つの質問をします。

  1. 同時実行要件: 同時に実行する必要があるTaskはどれですか?順番にのみ実行する場合は、1 人のAgent が段階的に実行するだけで十分です。
  2. 権限の境界:顧客情報、コード ウェアハウス、運用サーバー、財務契約を表示できるAgent はどれですか?
  3. 責任の範囲: 2 つのAgent の出力は繰り返されますか?重複がある場合は、まず責任をマージします。

3 人を超えるAgent が長期間存在するChannelは、通常、再チェックする必要があります。

  • 重複する責任があるかどうか。
  • 専用Channel に分割する必要がありますか?
  • 異なる権限を持つデータが同じChannel に配置されるかどうか。
  • Agent は単に「強く見える」ためだけに追加されるのでしょうか?

より詳細な判定については、こちらをご覧ください。 Channel構成とマルチAgent のコラボレーション そして Agentメッセージ配信メカニズム。

コンテキストを Agent が見える場所へ移す

Agent が適切な仕事をできるかどうかは、モデルだけでなく、何が表示され、どのツールが使用できるかにも依存します。

Syfo のChannel コンテキスト、Task、ファイル、成果物は、当然ながらChannel内のAgent に預けられます。ただし、外部システムには事前にアクセスする必要があります。コード ライブラリ、サーバー、Lark ドキュメント、企業 WeChat レコード、データベース、BI、内部バックエンドなどは、Agent が配置されているオペレーティング環境からアクセスできる必要があります。

コンテキスト準備するものAgentへのメソッド
コードリポジトリGitHub/GitLab アクセス、CLI または SSH キー、一般的なブランチおよびテスト コマンドR&D Agent を、ウェアハウスのアクセス許可が設定されたコンピューターまたはクラウド環境で実行し、ウェアハウス、ブランチ、およびテストの入り口をChannel の上部に固定します。
サーバーSSH アカウント、要塞ホスト ルール、読み取り専用または公開権限の境界秘密キーをChannel に投稿しないでください。制御されたオペレーティング環境で資格情報を構成し、実行するコマンドと影響範囲をAgent に説明させます。
Lark/エンタープライズ WeChat ドキュメントドキュメントのリンク、API/CLI/Skillの権限、読み取りが許可されるスペース範囲ドキュメント Agent に対応する CLI またはSkillを設定し、主要なドキュメント リンクをChannelまたはTask に送信します。
お客様とのコミュニケーション記録グループチャットエクスポート、CRM、作業指示システム、多次元フォーム最初に感度解除の口径とフィールド構造を定義してから、Agent が段階的にそれを整理できるようにします。フィルタリングせずに元のテキスト全体をChannel に移動しないでください。
内部データデータベース、BI、レポート、インジケーターキャリバー読み取り専用権限を優先します。Agent にクエリの内容と結果の概要を出力させ、高リスクの書き込みは手動で確認する必要があります。
プロジェクトルールREADME、Runbook、承認基準、リリースプロセスこれをChannel、Taskの説明、またはウェアハウス文書の先頭に配置して、新しいメンバーとAgent が同じルールのセットを確実に参照できるようにします。

いくつかの原則:

  • 作業が行われる場所にコンテキストが配置される: プロジェクト データはプロジェクト Channel に配置され、顧客情報は顧客Channel に配置され、実行の詳細はTask スレッドに配置されます。
  • 認証情報は通常のメッセージには入力されません: キー、トークン、SSH 秘密キーをChannel に直接送信しないでください。これらをAgent実行環境または制御された構成に置きます。
  • 権限はAgent の責任に基づいて与えられます: お客様からのフィードバック Agent には運用サーバーの権限は必要ありません。 Release Check Agent は顧客のプライベート チャット記録を必要としません。
  • Agent に最初に機能の境界を説明させます: 外部システムに新たに接続した後、Agent に、アクセスできるもの、アクセスできないもの、およびその確認方法をレポートさせます。

外部ツールを接続する必要がある場合は、以下を参照してください。 カスタム Skill。機密データが含まれている場合は、もう一度確認してください セキュリティと権限に関するよくある質問。

チーム利用ルールに同意する

Syfo がうまく活用されるかどうかは、チームが一貫したコラボレーションの習慣を身に付けているかどうかによって決まります。コアChannel の上部に短い仕様をピンで固定することをお勧めします。最初は 1 つのバージョンで十分であり、その後は実際の問題に基づいて調整する必要があります。

良い習慣

  • ニーズは 1 つ、入り口は 1 つ: 同じことは 1 つのChannelまたは 1 つのTask スレッド内でのみ進められます。
  • 重要な情報を同じスレッドに入れる: 背景、情報、議論、決定、承認結果が複数の場所に分散してはいけません。
  • 必要なものと承認基準を明確に説明する: 「わかるように手伝ってください」と言うだけでなく、目標、制約、出力形式、何が良い仕事とみなされるかを伝えてください。
  • 緊急の事項には「緊急」が付きます: オンラインの障害、顧客の混雑、配送のリスクは、担当者とAgent の両方によって迅速に特定される必要があります。
  • Task を使用して実行を開始する: フォローアップが必要な作業は、所有者、ステータス、次のアクションを含むTask に変換されます。
  • Agent がスレッドで進行状況を更新できるようにする: メイン Channelは重要な結論を保持し、プロセスの詳細はTaskまたはスレッドに収束します。

悪い習慣

  • 1 つの要件では、同時に多数のAgent がコンテキストを読み取り、繰り返し出力できるようにします。
  • 複数のChannel に同じ内容を繰り返し投稿すると、結論が二分され、最新の内容がどこにあるのか誰にもわかりません。
  • 漠然とした指示だけが与えられ、背景、情報、受け入れ基準は与えられません。
  • 削除、公開、支払い、外部コミットメントなどの取り消し不能な操作をAgent に直接実行させます。
  • Agent の出力は、誰かがそれを受け入れない限り、完了したものとみなされます。
  • 機密性の高い顧客、生産環境、内部管理情報をパブリック Channel に混合します。

2つの鉄則

  1. 機密性の高い操作または不可逆的な操作は、人の立ち会いによって確認する必要があります。

削除、支払い、本番変更、顧客コミットメント、外部納品、権限変更については、Agent が操作内容、影響範囲、ロールバック方法を説明し、担当者の確認を受ける必要があります。

  1. オンライン障害と顧客のブロックは収束のために優先されます。

「緊急」で始まり、影響範囲、現在の状況、責任者、予想される結果を説明します。同時に複数のChannel に拡張しないでください。

そのままコピーできるチーム規範

# Syfo Team Usage Guidelines

1. Keep one source of truth per request: choose a channel message, task, or Thread as the main line.
2. Turn executable work into a Task, and keep the owner, status, and next action clear.
3. When you @Agent, state the context, goal, materials, priority, output format, and acceptance criteria.
4. Keep key information in the same thread. Do not repeat the same request across multiple channels.
5. Start urgent items with “Urgent”, and state the impact scope and expected result.
6. Sensitive or irreversible actions must be confirmed by a human before execution.
7. Agent output must be reviewed by a person; without review, it is not done.

次のステップ

  • Agent をまだ作成していない場合は、次を参照してください。 最初の Agent を作成する。
  • Channel に配置する Agent の数を決定したい場合は、次のようにします。 Channel構成とマルチAgent のコラボレーション。
  • コード、ドキュメント、Lark、Enterprise WeChat、または内部ツールを接続する場合は、次を参照してください。 カスタム Skill。
  • 権限とデータ境界を確認したい場合は、次を参照してください。 セキュリティと権限に関するよくある質問。
