よくある質問

よくある質問

評価、オンボーディング、Agent の使用法、セキュリティ権限、ツール統合の問題。

Syfo を評価し、開始し、日常的に使用する際によくある質問を以下に示します。トピックごとに回答をすばやく見つけることができ、これらの質問を使用して、Syfo がどのチーム ワークフローに適しているかを判断することもできます。

1. Syfoと既存ツール

すでに Lark/WeChat Enterprise/Slack を持っていますが、なぜ Syfo が必要なのでしょうか?

Syfo では、すべての通信を移動する必要はありません。 「人とAgent が協力して作業を進める必要がある」Channel に適しています。

従来の IM では、AI はプラグインまたは別のウィンドウであることがよくあります。聞いてみると、自分でグループに戻らなければならず、Taskも自分で実行しなければならないことがわかりました。 Syfo との違いは、Agent がChannel内にあり、コンテキストを確認し、Task を受信し、ステータスを更新し、結果を配信できることです。

通常、Syfo が適している特性は次の 3 つです。

  • 議論は行動に変わります。
  • AI が実行に参加する必要があります。
  • プロセスと結果はチームに見える必要があります。

チームが Lark を使い続ける場合、Syfo はコマンド ライン ツールや統合機能を通じてドキュメント、グループ チャット、多次元テーブル、その他のコンテンツを読み書きできます。コラボレーション プロセスは Syfo Channel に残り、結果は必要に応じて既存のシステムに書き戻されます。

企業には多くの内部システムがありますが、Syfo はそれらをどのように通過できるのでしょうか?

すべての業務システムを Syfo に移行する必要はありません。より現実的なアプローチは、Syfo を既存のシステムと連携させることです。

短期的には、明確なツールの入り口から始めることができます。つまり、Agent に API、CLI (コマンド ライン ツール)、MCP (標準化されたツール接続プロトコル)、またはSkillを通じて必要な情報を読み取り、安定したアクションを実行させ、結果をChannel に返します。たとえば、ナレッジ ベースにクエリを実行し、BI インジケータを読み取り、作業指示を確認し、プロジェクト ステータスを同期します。

長期的には、アクセス許可に基づいて、より多くのシステム機能を徐々に開くことができます。重要なのは、各ツールに明確な境界線を持たせる必要があるということです。つまり、何を表示できるか、何を変更できるか、どの操作に人間の確認が必要かということです。

モデル呼び出しはどのChannel を使用しますか?データは安全ですか?

Syfo のモデル呼び出しは、OpenAI や Anthropic などの公式Channel、または大手メーカーの公式Channel を通じて行われます。負荷、フォールト トレランス、およびアカウンティングのために中間で使用されるトークン ゲートウェイは Syfo 自体によって構築され、未知のサードパーティ トークン Agent を経由しません。

なお、外部大規模モデルを呼び出している限り、リクエスト内容自体は対応するモデル提供者に送信されます。企業は、企業契約、組織の権限、および選択したモデル ベンダーのデータ分類に基づいて、どのコンテンツをAgent に引き渡すことができるかを決定する必要があります。

Syfo 自体が解決する必要があるのは、Agent が組織内で何を確認できるか、どのシステムにアクセスできるか、どのアクションを確認する必要があるか、およびこれらのプロセスを監査できるかどうかです。

2. Agent の作成と構成

Agent とは何ですか?

Agent は Syfo の AI メンバーです。これには名前、アバター、@ハンドルがあり、Channel に入り、表示されているコンテキストを読み取り、Task を要求し、スレッド (トピック、メッセージの下のサブ会話) の進行状況を更新できます。

Agent の作成方法がわかりません。どこから始めればよいですか?

空のプロンプトから始めないでください。まず実際のシナリオを選択し、次に逆算してどのChannel、Agent、ツールが必要かを判断します。

一般的な開始点は次のとおりです。

  • R&D コラボレーション: デマンド Channel、開発Agent、レビュー Agent。
  • 顧客サポート: 作業指示Channel、FAQ Agent、アップグレード処理Agent。
  • 調査分析: 情報ソース Channel、調査Agent、レポート Agent。
  • コンテンツ制作: トピック選択Channel、スクリプト Agent、エディター Agent。

まだよくわからない場合は、まず読んでください チーム導入ガイドの場合、設計はChannel の分割、Agent の数、コンテキストの移行から始まります。

新しいAgent をいつ作成し、いつ再利用するか?

同様のTaskのみがオブジェクトを変更する場合は、最初にAgent が再利用されます。

責任の範囲、権限、外部システム、機密データが明らかに異なる場合は、新しいAgent を作成します。たとえば、「顧客フィードバックの収集」と「本番環境のリリース検査」を同じAgent内に混在させることはできません。

経験則: 最初に 1 つのAgent を使用してみて、責任が実際に分割された後でAgent を分割します。

詳しい説明を参照 Channel とコラボレーション。

ローカルコンピュータとSyfoクラウドの違いは何ですか?

ローカル コンピューターは、開発、デバッグ、および一時的なTask に適しています。ローカル マシン上の既存のウェアハウス、コマンド ライン ツール、ログイン状態にアクセスできますが、コンピュータがシャットダウンされたり、ネットワークが切断されたりすると、Agent は動作を継続できません。

Syfo Cloud は、スケジュールされたTask、長期オンライン、チーム共有、より安定した実行環境に適しています。 24 時間自動的に実行する必要があるAgent は、まずクラウドに配置する必要があります。

私のコンピュータにはクロード コード/コーデックスがインストールされ、共通Skill/MCP が設定されています。 Syfo にアクセスするにはどうすればよいですか?

次の 2 つの状況があります。

ローカル Agent (コンピューターは Syfo に接続されています)

このコンピュータにAgent を作成すると、このコンピュータにインストールされているSkillと MCP が自動的に検出されます。追加のアクションは必要ありません。Agent は独自の環境で実行され、クロード コード、コーデックス、設定した CLI およびログイン ステータスを自然に使用できます。

Syfo Cloud Agent

クラウド Agent は独立した環境で実行され、コンピュータ上の構成を自動的に継承しません。Agent と非公開で直接チャットし、Skill/MCP のアドレスを指定して、インストールするように依頼します。次に例を示します。

Please install this MCP: https://example.com/my-mcp

インストールの進行状況と結果については、Agent が会話で返答します。詳しい説明を参照 カスタム Skill。

インストールされているSkillを表示する

で Agentページ→Skill タブ ページでは、現在のAgent にインストールされているSkill/MCP を確認できます。

Agent はブラウザのみに依存してページをクリックしますか?

いいえ。ブラウザの自動化は最後の手段にすぎません。

運用環境の場合、Agent に API、CLI (コマンド ライン ツール)、MCP (ツール接続プロトコル)、またはSkillを使用させることが最善です。このようにして、テストと監査が可能であり、ページ ボタンが変更されたからといって失敗することはありません。

業務システムにWebページのみでインターフェースがない場合は、最初はブラウザ自動化による検証も可能ですが、長期的には安定した動作をSkillやAPIに蓄積していく必要があります。

3. Channelとコラボレーション

Channel には何人のAgent を配置する必要がありますか?

最初は減らしてから増やしてください。固定プロジェクトの場合は、最初に「複数人 + 1 人の常駐Agent」を使用してプロセスを実行することをお勧めします。さらに多くの役割が必要な場合は、責任に応じて追加できます。詳しい判断については、 Channel とコラボレーション。

複数のAgent が混乱を引き起こすでしょうか?

はい、最初から複数のAgent を同じChannel に引き込まないでください。まず、誰がTask を解体するか、誰が実行するか、誰がレビューするか、そしてどのアクションに確認が必要かを定義します。詳細な方法を参照してください。 Channel とコラボレーション。

スケジュールされたTaskはAgent に割り当てられていますが、Channel内のすべてのAgent が通知を受信したのはなぜですか?

通常の状況では、スケジュールされたTaskは特定のAgent に送信され、対応するトピックまたはTaskのコンテキストに配置される必要があります。正確な配信はターゲット Agent にのみ通知します。ChannelレベルのメッセージのみがChannel内のAgent にブロードキャストされます。

他のAgent の邪魔をしたくない場合:

  • リマインダーまたはスケジュールされたTask を作成するときは、実行Agent を明確に指定します。
  • 以降のディスカッションは同じTaskまたはトピック内で行います。
  • 実装の詳細を一般的なChannel メッセージで補充しないでください。

Agent が割り当てられていて、フルChannel ブロードキャストがまだトリガーされていることを確認した場合は、リマインダー ID、Channel、時間、および通知を受信したAgent のリストを保存し、トラブルシューティングのためにチームに任せることができます。

Agent が後続の質問を同じTask に分類するのはなぜですか?

同じスレッドまたは同じTask コンテキストで質問を続けると、Agent は最初に同じものを追加していると認識します。

新しいTask を開始するには、メイン Channel に戻って新しい @メンションを投稿するか、「これは新しいTaskです」と明示的に言います。

4. Agent の動作とチューニング

Agent を動作させるにはどうすればよいですか?

Channel内で直接 @メンションし、目標、入力、出力形式、受け入れ基準を明確に述べます。

@Research-Assistant summarize the product issues customers mentioned in this channel this week, classify them by P1/P2/P3, and suggest next actions.

これがフォローアップが必要なジョブの場合、Agent はメッセージをTask に変換し、実行を開始します。

Agent は動作していることを示し続けますが、応答しません。どうすればいいですか？

まだ繰り返し @ しないでください。

次の 3 つの状況が考えられます。

  • Taskは非常に長く、まだキューに入れられているか処理中です。
  • Agent の実行環境が停止しているため、セッションをリセットする必要があります。
  • ローカル ランタイム、ネットワーク、またはツールの構成に問題があります。

まず、Agent のアクティビティ ステータスを確認できます。長期間進捗がない場合は、管理者にAgent をリセットするか、対応するコンピュータを再起動するように依頼してください。

Agent が 1 ～ 2 分待たなければならない場合があるのはなぜですか?どこが遅いのでしょうか？

Agent は永続的なセッションであり、毎回新しい空のセッションを開くわけではありません。複数のChannelと複数のTaskで同時に動作する場合があります。前のコマンド、ツール呼び出し、または長いTaskが終了していない場合、後続のメッセージはキューに入れられます。

これを高速化するには、分業を狭めることが重要です。

  • ビジネスまたはモジュールごとにChannel を分割し、1 人のAgent があまりにも多くのトピックにまたがらないようにします。
  • Agent は最初に 1 ～ 2 つのモジュールに焦点を当てます。
  • セッションが長時間実行され、コンテキストが長くなった後、アイドル状態になったときにセッションをリセットします。

Agent が長期間非アクティブだった場合は、Agent をリセットするかどうかを決定する前に、まずアクティビティ ページとTaskのステータスを確認できます。

問題が発生すると、Agent は常に停止して確認を待つのはなぜですか?

Agent自体は、独立してログを取得し、エラーを読み取り、修復を試みることができます。いつも立ち止まってあなたを待っています。通常、これは能力の問題ではなく、作業の境界が明確に合意されていない、または現在のセッションが長すぎる、または忙しすぎることが原因です。

ルールをAgent の長期メモリに書き込むことをお勧めします。

  • ログの確認、テストの再実行、明らかなエラーの修正などのリスクの低い操作は、独立して実行できます。
  • 運用変更、データ削除、リリース、権限変更、外部コミットメントなどの高リスク操作は、2 回確認する必要があります。
  • 確認の際は、変更の範囲、影響、ロールバック方法を明確にしてください。

それでも頻繁にフリーズする場合は、ChannelとTaskの範囲を狭めて、無関係なコンテキストを処理する必要が少なくなるようにします。

Agent がブランチを作成し、小さな変更のためにウェアハウス全体を繰り返し読み取るのは普通のことですか?

これは通常、作業仕様がまだ調整されていないためであり、このように使用する必要はありません。

Agent に次のように明示的に伝えることができます。

  • 小規模なコピーライティング、小規模な構成、および単一ファイルの修復は、現在のブランチで直接変更できます。
  • 中程度以上の変更、または複数の人が並行して作業している場合は、機能ブランチを使用します。
  • 複数のAgent が同じウェアハウスを同時に変更する場合は、git worktree を使用します。
  • 毎回倉庫全体をスキャンしないでください。まずTaskごとに関連ファイルを見つけて、必要に応じて範囲を拡張します。

これらの設定をAgent の長期メモリに書き込むと、動作がより安定します。

Agent が間違いを犯した場合はどうすればよいですか?

スレッド内で問題を直接指摘し、新しい制約を与えます。

This direction is not right. Please keep only the issues customers explicitly mentioned, and do not add your assumptions.

外部送信、削除、本番環境の変更、支払い、人員などのリスクの高いアクションを含む場合は、Agent に確認情報を提供し、承認後に実行する必要があります。

Agent を一時停止、削除、またはリセットするにはどうすればよいですか?

Agent がChannel から削除されると、そのChannel からメッセージを受信しなくなります。

Agent が長時間停止している場合は、資格のある管理者がセッションをリセットするか、実行環境を再起動できます。一般メンバーをリセットできるかどうかは、組織およびAgent の権限設定によって異なります。

5. セキュリティと権限

Agent は私の個人データにアクセスできますか?

Agent は、招待されたChannelと自身のプライベート チャットにのみアクセスできます。参加していないChannel を表示することも、他のメンバー間のプライベート メッセージを表示することもできません。外部システムにも追加の認証が必要です。

Agent はパーソナル アシスタントのようにスケジュール、ドキュメント、Larkのメッセージを処理できますか?

はい、ただし、ID と権限は個別に設計する必要があります。パーソナル アシスタント Agent には個人情報が含まれるため、複数人で共有するのには適していません。各人が個別のAgent を作成するか、機密性の高いポジションに対して独立したAgent を構成することをお勧めします。

どのAgent ともプライベートにチャットできますか?

デフォルトでは不可能です。

Agent が作成されたら、デフォルトでは、作成者（所有者）のみがプライベートにチャットできます。これは、オープンなプライベートチャットを通じて機密情報が漏洩するのを防ぐためです。

組織の他のメンバーがAgent とプライベートにチャットできるようにしたい場合は、次のことを行う必要があります。 Agent のオーナー 設定でオンにします。

  1. 左列の「メンバー」→対応するAgent を選択
  2. 「プライベートメッセージ設定」を開きます
  3. 「他の人がこのAgent にプライベート メッセージを送信できるようにする」をオンにします
  4. 「指定メンバー」または「ビジネスオーナー」を選択して保存します

Agent とプライベートにチャットできない場合は、所有者がまだ許可を与えていないことを意味します。まずChannel内で @ it と共同作業するか、Agent の所有者に連絡してプライベート メッセージ設定を調整することができます。

Agent はどのChannel を見ることができますか?

Agent は、招待されたChannel のみを表示できます。

Channel に入っていないAgent は、Channel のメッセージ、スレッド、Task、添付ファイルを表示できません。プライベート Channel には明示的な招待が必要です。招待されていない人もAgentもそれらを見ることができません。

Agent は他の人のプライベート チャットを見ることができますか?

いいえ。

Agent はメンバー間の DM を見ることができません。参加している DM と、参加しているChannel のコンテンツのみを表示できます。

公的機関が顧客データを漏洩する可能性はありますか?

同じ公的Agent を複数の機密性の高い顧客Channel に配置しないでください。

最も一般的な懸念は、プロジェクト A のAgent が取得した情報がプロジェクト B で参照されるかどうかです。顧客や機密性の高い境界を越えて同じAgent を共有することを避けるために、顧客、部門、または機密性の高い立場に応じてAgent とChannel を個別に作成することをお勧めします。

Agent がプライベート Channelへの参加を制限されているのはなぜですか?

Agent は通常のロボットではないため、コンテキストを読み取り、Task を実行できます。

プライベート Channel には通常、顧客、財務、契約、研究開発、人事情報が含まれます。Agent を追加する前に、Agent の責任者、Agent が表示できる内容、プライベート チャットができるかどうか、他の人から呼び出せるかどうかを確認する必要があります。

Agent とプライベートにチャットできるのは誰ですか?

Agent所有者のプライベート メッセージ設定によって異なります。

Agent が作成されたら、デフォルトでは、所有者のみがプライベートにチャットできます。所有者は、「メンバー → Agent の選択 → プライベート メッセージ設定」で「他の人がこのAgent にプライベート メッセージを送信できるようにする」をオンにし、「指定したメンバー」または「会社所有者」を指定できます。関連する手順については、このページの前半にあるプライベート チャットの質問を参照してください。

公的Agent をより多くの人に公開できるようになります。機密データに関係するAgent は、プライベート チャットの範囲を制限するか、Channel内でのコラボレーションのみを許可する必要があります。

Agent が外部システムを使用する場合、認証情報はどこに配置されますか?

Agent が実行されている場所によって異なります。

Agent がローカル マシン上で実行されている場合、ローカル マシン上の既存の CLI、ログイン状態、および設定ファイルを使用できます。Agent が Syfo Cloud で実行される場合、アクセスできるツールと認証情報はホスティング環境で個別に設定する必要があります。

これが望ましい権限の境界であることを確認しない限り、複数の機密性の高いAgent が同じ外部システムのログイン状態を共有することを許可しないでください。

リスクの高い作戦中にAgent の混乱を避けるにはどうすればよいですか?

リスクの高い操作を制御するために、「Agent が理解するという信頼」に依存しないでください。いつ確認するか、誰に確認するか、何を確認するか、何を保持するかという二次確認のルールを事前に書き留めておく必要があります。システム内での確認が必要な場合、Agent にアクションカード（操作確認カード - Agent生成後、実行前にクリックして承認する必要がある）の生成を求めることができます。詳細な手順については、を参照してください。 ベストプラクティス。

どのアクションを手動で確認する必要がありますか?

Agent に次のアクションを直接自動的に実行させないでください。

  • 顧客との約束、電子メール、お知らせ、ソーシャル メディア コンテンツを外部に送信します。
  • 支払い、返金、補償、契約関連の行為。
  • 本番環境のリリース、データ移行、データ削除。
  • 担当者、権限、アカウント、およびキーに関連する変更。
  • 失敗後にロールバックすることが困難な操作。

Agent に操作指示と影響範囲を先に伝え、確認してから実行させます。

6. ツールと拡張機能

Agent はスケジュールに基づいてTask を自動的に実行できますか?

はい。Agent に時間とアクションを直接伝えます。

@Research-Assistant every morning at 9, summarize yesterday's discussion in #product-team and post the briefing here.

スケジュールされたTaskでは、Agent の実行環境が対応する時間に利用可能である必要があります。長期安定実行が必要な場合はSyfo Cloudのご利用をおすすめします。

ファイルと成果物の違いは何ですか?

ドキュメントとは、スクリーンショット、ログ、参考資料などのプロセス資料です。

成果物とは、レポート、MR、導入アドレス、最終 PDF などの最終結果です。ファイルをアップロードしても自動的に成果物にはならないため、明示的にタグ付けするか、Task にリンクする必要があります。

「ファイルと成果物の違いは何ですか?」という質問を参照してください。このページで。

Syfo Agent には GPT 画像などの画像/ビデオ生成機能がありますか?

ローカルのクロード コード/コーデックス設定に従ってください。デフォルトではなし。 Syfo Agent には画像またはビデオ生成モデルが付属していないため、関連するSkillまたは CLI ツールを自分で設定する必要があります。

推奨される設置方法 Kamay.ai（AIクリエイティブ生成・素材制作ツール）。対応するSkill/CLI をインストールするようにAgent に直接依頼してください。 API キーの認証が完了すると、SOTA モデルを使用して画像やビデオを生成できるようになります。

Follow the Kamay CLI Skill installation guide (https://raw.githubusercontent.com/reorc/kamay-cli-skill/refs/heads/main/references/INSTALL.md) to install and configure the Kamay CLI Skill.

開発プロセスはブラックボックスのようなものです。結果だけを見ることができます。何が変更されたのか、またロールバックする方法はどのようにしてわかりますか?

この段階では、「デュアル スクリーン」で作業することをお勧めします。

  • Syfo の会話とAgent のアクティビティ ページを 1 つの画面に表示して、どのツールが呼び出されているか、どのコマンドが実行されているかを理解します。
  • 1 つの画面で VS Code またはターミナルを開いて、git diff、テスト結果、コミット レコードを表示します。

バージョン管理とロールバックは git によって処理されます。Agent が終了したら、どのファイルが変更されたか、変更された理由、およびそれらを確認する方法を説明させます。ロールバックしたい場合は、git diff、commit、または worktree に基づいて復元させます。

プライベート展開はサポートされていますか?

議論できる。

企業が製品 UI、通信ゲートウェイ、データ ストレージ、およびAgent の動作環境をすべて自社のインフラストラクチャ内に置く必要がある場合は、営業担当者に連絡してプライベート展開ソリューションを評価できます。
