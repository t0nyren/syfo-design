常见问题

常见问题

评估、上手、Agent 使用、安全权限和工具集成问题。

这里整理了评估、上手和日常使用 Syfo 时最常见的问题。你可以按主题快速找到答案，也可以从这些问题判断 Syfo 适合放进哪些团队工作流。

一、Syfo 和现有工具

我们已经有飞书 / 企业微信 / Slack，为什么还要 Syfo？

Syfo 不是要求你把所有沟通都迁走。它更适合放那些“人和 Agent 要一起推进工作”的频道。

传统 IM 里，AI 往往是一个插件或另一个窗口。你问完以后，结果要自己搬回群里，任务也要自己跟。Syfo 的区别是：Agent 就在频道里，看得到上下文，可以接任务、更新状态、交付结果。

适合用 Syfo 的地方通常有三个特征：

  • 讨论会变成行动；
  • 需要 AI 参与执行；
  • 过程和结果需要被团队看见。

如果团队还继续使用飞书，Syfo 可以通过命令行工具或集成能力读写文档、群聊、多维表格等内容；协作过程留在 Syfo 频道里，必要时再把结果写回现有系统。

企业内部系统很多，Syfo 怎么打通？

不需要把所有业务系统都搬进 Syfo。更现实的方式是让 Syfo 和现有系统配合工作。

短期可以从明确的工具入口开始：让 Agent 通过 API、CLI（命令行工具）、MCP（一种标准化的工具连接协议）或 Skill 读取必要信息、执行稳定动作，再把结果回到频道里。比如查询知识库、读取 BI 指标、检查工单、同步项目状态。

长期可以按权限逐步开放更多系统能力。关键是每个工具都要有清晰边界：能看什么、能改什么、哪些动作需要人确认。

模型调用走什么渠道？数据安不安全？

Syfo 的模型调用会走 OpenAI、Anthropic 等官方渠道，或经过大厂正规渠道。中间用于负载、容错和计费的 token 网关由 Syfo 自建，不会经过不明第三方 token 代理商。

需要注意的是：只要调用外部大模型，请求内容本身就会发送给对应模型供应商。企业应按所选模型供应商的企业协议、组织权限和数据分级来决定哪些内容可以交给 Agent 处理。

Syfo 自身要解决的是：Agent 在组织内能看到什么、能访问什么系统、什么动作必须确认，以及这些过程能不能被审计。

二、创建和配置 Agent

Agent 是什么？

Agent 是 Syfo 里的 AI 成员。它有名字、头像和 @handle，可以进入频道、读取可见上下文、认领任务、在 Thread（话题，消息下的子对话）里更新进度。

不知道怎么创建 Agent，应该从哪里开始？

不要从空白 prompt 开始。先选一个真实场景，再反推需要哪些频道、Agent 和工具。

常见起点包括：

  • 研发协作：需求频道、开发 Agent、Review Agent；
  • 客户支持：工单频道、FAQ Agent、升级处理 Agent；
  • 研究分析：信息源频道、Research Agent、Report Agent；
  • 内容生产：选题频道、Script Agent、Editor Agent。

如果还不确定，可以先看 团队落地指南，从频道拆分、Agent 数量和上下文迁移开始设计。

什么时候新建 Agent，什么时候复用 Agent？

如果只是同类任务换了对象，优先复用 Agent。

如果职责、权限、外部系统、敏感数据范围明显不同，就新建 Agent。比如“客户反馈整理”和“生产环境发布检查”就不该混在一个 Agent 里。

经验法则：先用一个 Agent 试，等职责真的分化后再拆。

更多说明见 频道与协作。

本地电脑和 Syfo Cloud 有什么区别？

本地电脑适合开发、调试、临时任务。它能访问你本机已有的仓库、命令行工具和登录态，但电脑关机或网络断开时 Agent 就不能持续工作。

Syfo Cloud 适合定时任务、长期在线、团队共享和更稳定的执行环境。需要 24 小时自动跑的 Agent，优先放云端。

我的电脑已经安装了 Claude Code / Codex 并配置了常见 Skill / MCP，如何接入 Syfo？

分两种情况：

本机 Agent（电脑已接入 Syfo）

在这台电脑上创建 Agent 后，本机已安装的 Skill、MCP 会自动被发现，无需额外操作。Agent 跑在你自己的环境里，天然能用到 Claude Code、Codex 以及你已配好的 CLI 和登录态。

Syfo Cloud Agent

云端 Agent 跑在独立环境里，不会自动继承你电脑上的配置。直接私聊 Agent，给出 Skill / MCP 的地址，要求它安装即可，例如：

请安装这个 MCP：https://example.com/my-mcp

安装进度和结果 Agent 会在对话里回复你。更多说明见 自定义 Skill。

查看已安装技能

在 Agent 页面 → Skill 标签页，可以看到当前 Agent 已经安装了哪些 Skill / MCP。

Agent 是不是只能靠浏览器点页面？

不是。浏览器自动化只是最后手段。

生产环境最好让 Agent 使用 API、CLI（命令行工具）、MCP（工具连接协议）或 Skill。这样可测试、可审计，也不会因为页面按钮改了就失败。

如果业务系统只有网页、没有接口，可以先用浏览器自动化做验证，但长期应该把稳定动作沉淀成 Skill 或 API。

三、频道和协作

一个频道里该放几个 Agent？

先少后多。固定项目建议先用“多人 + 1 个驻场 Agent”跑通流程；需要更多角色时，再按职责增加。详细判断见 频道与协作。

多个 Agent 会不会乱？

会，所以不要一开始就把多个 Agent 拉进同一个频道。先定义谁拆任务、谁执行、谁 review、哪些动作需要人确认；详细方法见 频道与协作。

定时任务已经指派给某个 Agent，为什么全频道 Agent 都收到通知？

正常情况下，定时任务应该指向具体 Agent，并放在对应话题或任务上下文里。精准投递只通知目标 Agent；频道级消息才会广播给频道里的 Agent。

如果你想避免打扰其他 Agent：

  • 创建提醒或定时任务时，明确指定执行 Agent；
  • 把后续讨论放在同一个任务或话题里；
  • 不要用普通频道消息反复补充执行细节。

如果确认已经指定了 Agent，仍然触发全频道广播，可以保留提醒 ID、频道、时间和收到通知的 Agent 列表，交给团队排查。

Agent 为什么会把后续问题归到同一个任务？

如果你在同一个 Thread 或同一个任务上下文里继续追问，Agent 会优先认为你在补充同一件事。

要开新任务，回到频道主流里发一条新的 @mention，或者明确说“这是一个新任务”。

四、Agent 行为和调优

怎么让 Agent 开始工作？

在频道里直接 @mention，并说清楚目标、输入、输出格式和验收标准：

@Ringo 整理这个频道里本周客户提到的产品问题，按 P1/P2/P3 分类，并给出建议动作。

如果这是一个需要跟进的工作，Agent 会把消息转成任务并开始执行。

Agent 一直显示工作中、不回复，怎么办？

先不要反复 @ 它。

可能的情况有三种：

  • 任务确实很长，还在排队或处理中；
  • Agent 运行环境卡住，需要重置 session；
  • 本地 runtime、网络或工具配置有问题。

可以先看 Agent 的活动状态。如果长时间无进展，找管理员重置 Agent 或重启对应电脑。

为什么 Agent 有时要等一两分钟？慢在哪？

Agent 是常驻会话，不是每次都新开一个空白对话。它可能同时在多个频道、多个任务里工作；当前一条命令、工具调用或长任务还没结束时，后续消息会排队。

想让它更快，关键是把分工收窄：

  • 按业务或模块拆频道，不要让一个 Agent 横跨太多主题；
  • 一个 Agent 优先专注 1–2 个模块；
  • 会话跑久、上下文变长后，在空闲时重置 session。

如果 Agent 长时间没有活动，可以先看活动页和任务状态，再决定是否重置。

Agent 遇到问题为什么总停下来等我确认？

Agent 本身可以自主拉日志、看错误、尝试修复。它总停下来等你，通常不是能力问题，而是工作边界没有约定清楚，或者当前会话太长、太忙。

建议把规则写进 Agent 的长期记忆：

  • 低风险操作，例如查看日志、重跑测试、修复明显报错，可以自主推进；
  • 高风险操作，例如改生产、删数据、发布、权限变更、对外承诺，必须二次确认；
  • 确认时要说清变更范围、影响面和回滚方式。

如果它仍然频繁停住，就把频道和任务范围再收窄，让它少处理无关上下文。

Agent 一个小改动就建分支、反复通读整个仓库，正常吗？

这通常是工作规范还没调好，不是必须这样用。

你可以明确告诉 Agent：

  • 小文案、小配置、单文件修复，可以直接在当前分支改；
  • 中等以上改动，或者多人并行时，用 feature branch；
  • 多个 Agent 同时改同一仓库时，用 git worktree；
  • 不要每次都全仓扫描，先按任务定位相关文件，必要时再扩大范围。

把这些偏好写进 Agent 的长期记忆后，行为会稳定很多。

Agent 做错了怎么办？

在 Thread 里直接指出问题，并给出新的约束：

这个方向不对。请只保留客户明确说过的问题，不要加入你的推测。

如果涉及外部发送、删除、生产环境变更、支付、人事等高风险动作，应该让 Agent 先给出确认信息，由人批准后再执行。

如何暂停、移除或重置 Agent？

从频道移除 Agent 后，它不再接收该频道消息。

如果 Agent 长时间卡住，可以由有权限的管理员重置 session 或重启运行环境。普通成员是否能重置，取决于组织和 Agent 权限设置。

五、安全和权限

Agent 可以访问我的私人数据吗？

Agent 只能访问它被邀请进入的频道和它自己的私聊。它看不到没有加入的频道，也看不到其他成员之间的私信。外部系统还需要额外授权。

Agent 能不能像个人助理一样处理日程、文档、飞书消息？

可以，但要单独设计身份和权限。个人助理型 Agent 涉及个人信息，不适合多人共用。建议每人单独创建 Agent，或为敏感岗位配置独立 Agent。

我可以私聊任何 Agent 吗？

默认不能。

Agent 创建后，默认只有创建者（owner）可以私聊。这是为了防止敏感信息通过开放式私聊泄露。

如果你希望组织内其他成员也能私聊某个 Agent，需要由 Agent 的 owner 在设置里开启：

  1. 左侧栏「成员」→ 选择对应 Agent
  2. 打开「私信设置」
  3. 开启「允许他人私信此 Agent」
  4. 选择「指定成员」或「企业所有人」，保存

如果你无法私聊某个 Agent，说明 owner 还未对你开放权限。可以先在频道里 @ 它协作，或联系 Agent 的 owner 调整私信设置。

Agent 能看到哪些频道？

Agent 只能看到它被邀请进入的频道。

没有进频道的 Agent 看不到频道消息、Thread、任务和附件。私有频道需要显式邀请；没有被邀请的人和 Agent 都看不到。

Agent 能看到其他人的私聊吗？

不能。

Agent 看不到成员之间的 DM。它只能看到自己参与的 DM，以及自己所在频道里的内容。

公用 Agent 会不会泄露客户数据？

不要把同一个公用 Agent 放进多个敏感客户频道。

最常见的顾虑是：一个 Agent 在 A 项目中获取的信息，会不会在 B 项目中被引用。建议按客户、部门或敏感岗位分别创建 Agent 和频道，避免跨客户、跨敏感边界共用同一个 Agent。

Agent 加入私密频道为什么要受限制？

因为 Agent 不是普通机器人，它能读取上下文并执行任务。

私密频道通常包含客户、财务、合同、研发或人事信息。把 Agent 加进去前，需要确认谁是 Agent 的负责人、它能看什么、能不能私聊、能不能被其他人调用。

谁可以私聊 Agent？

取决于 Agent owner 的私信设置。

Agent 创建后，默认只有 owner 可以私聊。owner 可以在「成员 → 选择 Agent → 私信设置」里开启「允许他人私信此 Agent」，并指定「指定成员」或「企业所有人」。相关说明可以参考本页前面的私聊问题。

公用 Agent 可以开放给更多人；涉及敏感数据的 Agent 应限制私聊范围，或只允许在频道内协作。

Agent 使用外部系统时，凭据放在哪里？

取决于 Agent 运行在哪里。

如果 Agent 跑在你的本机，它能使用本机已有的 CLI、登录态和配置文件。 如果 Agent 跑在 Syfo Cloud，需要在托管环境里单独配置它能访问的工具和凭据。

不要让多个敏感 Agent 共用同一套外部系统登录态，除非你确认这就是想要的权限边界。

高风险操作怎么避免 Agent 乱来？

高风险操作不要靠“相信 Agent 会懂”来控制，要提前写清二次确认规则：什么时候确认、谁确认、确认什么、保留什么。需要系统内确认时，可以让 Agent 生成 Action Card（操作确认卡片——Agent 生成后需要人点击批准才执行）。详细说明见 最佳实践。

哪些动作必须人工确认？

以下动作不要让 Agent 直接自动执行：

  • 对外发送客户承诺、邮件、公告、社媒内容；
  • 支付、退款、补偿、合同相关动作；
  • 生产环境发布、数据迁移、删除数据；
  • 人事、权限、账号、密钥相关变更；
  • 任何失败后难以回滚的操作。

让 Agent 先给出操作说明和影响范围，由人确认后再执行。

六、工具和扩展

Agent 可以定时自动执行任务吗？

可以。直接对 Agent 说时间和动作：

@Ringo 每天早上 9 点，把昨天 #dev 频道的讨论整理成简报发到这里。

定时任务需要 Agent 的运行环境在对应时间可用。需要长期稳定执行时，建议使用 Syfo Cloud。

文件和交付物有什么区别？

文件是过程材料，例如截图、日志、参考文档。

交付物是最终成果，例如报告、MR、部署地址、最终 PDF。上传文件不会自动变成交付物，需要明确标记或挂到任务上。

可以参考本页“文件和交付物有什么区别？”这一问。

Syfo Agent 有没有图片 / 视频生成能力，比如 GPT Image？

跟随你本机 Claude Code / Codex 的配置，默认没有。Syfo Agent 不会自带图像或视频生成模型，需要你自行配置相关的 Skill 或 CLI 工具。

推荐安装 Kamay.ai（AI创意生成和素材制作工具）。直接要求你的 Agent 安装对应的 Skill / CLI，完成 API Key 认证后即可使用 SOTA 模型生成图片和视频。

根据文档说明 Kamay CLI Skill 安装配置指南 (https://raw.githubusercontent.com/reorc/kamay-cli-skill/refs/heads/main/references/INSTALL.md) 安装配置 Kamay CLI 技能。

开发过程像黑盒，只能看结果，怎么知道改了什么、怎么回滚？

现阶段建议用“双屏”工作：

  • 一屏看 Syfo 对话和 Agent 活动页，了解它正在调用什么工具、执行什么命令；
  • 一屏打开 VS Code 或终端，看 git diff、测试结果和提交记录。

版本和回滚交给 git 承接。Agent 做完后，让它说明改了哪些文件、为什么改、怎么验证；如果要回滚，就让它基于 git diff、commit 或 worktree 恢复。

是否支持私有化部署？

可以讨论。

如果企业要求产品 UI、通信网关、数据存储和 Agent 运行环境都在自己的基础设施里，可以联系销售评估私有化部署方案。
