Prácticas de equipo

Deje que los Agents se conviertan realmente en miembros del equipo

Experiencia práctica desde planificación, Agents de clasificación, gestión de contexto hasta mantenimiento del tablero de Tasks.

Las siguientes prácticas son adecuadas para personas que han comenzado a probar Syfo y están listas para incorporar Agent al trabajo diario del equipo. Primero puede observar la división de Channels y la cantidad de Agents.Guía de inicio del equipoyComposición de Channels y colaboración entre múltiples Agents, aquí nos centramos en algunos hábitos que son más fáciles de evitar y que pueden mejorar la eficiencia en la colaboración diaria.

En una frase:La persona da las metas y principios y el plan se le entrega al Agent. No estás operando una herramienta, sino liderando un equipo que crecerá por sí solo, centrándose en los requisitos, el juicio y la aceptación.

Deja que el Agent te ayude a planificar

No tiene que descubrir cómo configurarlo usted mismo, deje la planificación en manos del Agent:

  • Cuéntele a un Agent sobre su negocio, proceso y estructura organizacional, y pídale que le sugiera: cómo dividir los Channels, qué Agents colocar en cada Channel y cuántos Agents deben asignarse a cada tarea.
  • Dados los objetivos y limitaciones (como "Es necesario reconstruir N sistemas, cuántos días, cuántas personas y qué recursos"), producirá un plan e incluso dará la función y el aviso de cada Agent. Puedes crear uno nuevo y pegarlo para comenzar.
  • Si hay alguna desviación, se puede calibrar en cualquier momento. Se escribirá en la memoria a largo plazo y cuanto más se use, más se adaptará al equipo.

Este es un paso crítico desde "operar la herramienta usted mismo" hasta "traer un equipo de Agents".

Triaje de diseño/Agent residente

Cuando se produce colaboración multicanal, normalmente se requiere un Agent "residente" para la distribución y coordinación de la información (como el mostrador de orientación de un hospital). Existen varios principios para hacer un buen uso de él:

  • Responsabilidad única: El Agent en el sitio solo realiza desvío y coordinación, pero no realiza desarrollo específico. Una vez que se estanque en los detalles del desarrollo, no habrá tiempo para una clasificación oportuna y habrá colas.
  • Gobernado por el creador: Sus principios de funcionamiento son entrenados y restringidos por el propietario, y otros no pueden modificarlos a voluntad; al igual que un colega que tiene un gerente directo, la retroalimentación se brinda uniformemente al gerente.
  • Dar un marco, no acumular reglas: No hagas cientos de reglas. Junto con él, abstraemos el "marco de juicio + reglas de enrutamiento" y lo dejamos funcionar de acuerdo con el marco. Si se desvía, recalibrarlo.
  • Forme un grupo para la toma de decisiones entre departamentos.: La toma de decisiones que involucra a múltiples departamentos permite que los Agents de clasificación y los tomadores de decisiones humanos en cada departamento discutan en el mismo Channel. Después de llegar a un acuerdo, cada departamento devolverá la tarea a su propio departamento. No permita que un Agent realice la clasificación en dos sistemas grandes.

Gestione el contexto y ahorre dinero

El Agent es una sesión residente y el contexto se acumulará y se ralentizará después de ejecutarse durante mucho tiempo.

  • Restablecer durante el tiempo de inactividad: Restablezca las sesiones manualmente o en lotes cuando esté inactivo; la memoria a largo plazo no se perderá y el Agent se puede restaurar al nodo de trabajo anterior después de reiniciar. Restablecer interrumpirá las Tasks en curso, así que hágalo cuando lo desee.
  • ventana de contexto de control: Muchas Tasks no requieren contextos muy largos. Al controlar el contexto predeterminado dentro de un rango razonable (como 400K), se reducirán el costo y el retraso de una sola solicitud, y las mediciones reales básicamente no tienen ningún impacto en el rendimiento.
  • Seleccionar modelos por escena：
  • Codificación diaria, análisis de registros, informes técnicos, Tasks automatizadas: dar prioridad a GPT/Codex, que tiene menores costos y respuesta más rápida;
  • Frontend, UI, estética, redacción publicitaria: Claude generalmente se desempeña mejor.

Sólo cuando la división del trabajo se hace bien el Agent puede tener "tiempo libre" para reiniciar; de lo contrario, estará ocupado todo el tiempo y sólo podrá esperar hasta altas horas de la noche para quedar libre.

Escribir código con el Agent

  • Mira los resultados, mira menos las diferencias.: Cuando la capacidad del modelo es suficiente, en lugar de mirar el código línea por línea, es mejor mirar el efecto de ejecución final para su aceptación. Cuando desee ver cambios, use pantallas duales: una pantalla para hablar con el Agent y una pantalla para usar VS Code para ver git diff y confirmar registros; También puede ver las llamadas a herramientas y el razonamiento en la página de actividad del Agent.
  • Tarea independiente única de calidad de código: Además de la función, solicite periódicamente al Agent que divida archivos largos y limpie la capacidad de mantenimiento (los archivos individuales deben controlarse dentro de unos pocos cientos de líneas para evitar conflictos causados por varios Agents que modifican el mismo archivo); generalmente están de acuerdo en los estándares de codificación.
  • Use git para hacer múltiples versiones y árbol de trabajo para hacer paralelismo:git es la base para versiones y reversiones. Cuando varios Agents desarrollan el mismo almacén en paralelo, utilice el árbol de trabajo de git (solo dígaselo al Agent). El árbol de trabajo ocupa mucho espacio en el disco duro. Puede reutilizar los enlaces suaves dependientes instalados en el directorio raíz o eliminarlos después de su uso.

Acordar límites autónomos y confirmación secundaria.

El Agent tiene la capacidad de solucionar problemas de forma independiente (por ejemplo, si falla una implementación, extraerá el registro para repararlo). Si siempre se detiene y espera confirmación, probablemente sea porque los límites no están claramente acordados o el contexto es demasiado largo.

  • Explicar claramente las especificaciones de trabajo.: Las operaciones de bajo riesgo pueden realizarse de forma independiente (si puede leer los registros y resolver el problema usted mismo, no pregunte); Se reconfirman operaciones de alto riesgo.
  • Confirmación secundaria de alto riesgo(Operaciones irreversibles o de alto impacto como cambios de producción, eliminación de datos, liberaciones, compromisos externos y cambios de permisos):
  • cuando: Operaciones irreversibles o de alto impacto;
  • quien confirma: El líder humano de este módulo;
  • Confirmar que: Cambie el alcance, el área de impacto, el método de reversión y conserve las diferencias y los registros.
  • Después de ejecutar durante mucho tiempo y establecer la confianza, la confirmación se puede reducir para escenarios habituales y conservarse solo para situaciones de alto riesgo.

Deje que el Agent mantenga el tablero de Tasks

  • El tablero de Tasks se entrega al Agent para su mantenimiento, y la persona solo realiza solicitudes y verificaciones de aceptación.
  • Cuando el tiempo esté fragmentado, pídale directamente al Agent que haga un balance del tablero de Tasks (lo que no se ha avanzado, qué necesita actualizarse), o haga un balance de git (qué cambios en mi nombre no se han fusionado).
  • Si desea que una tarea quede "suspendida y no avanzada por el momento": utilice las dependencias (bloques) entre Tasks para expresar la secuencia. Cambiar el estado de la tarea notificará al Agent correspondiente. Este es un mecanismo de colaboración normal.
